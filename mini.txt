ABSTRACT

PetCloud is a smart pet-care web platform designed to simplify daily pet
management by integrating essential online features with an IoT-enabled automated
feeding system. The project aims to provide pet owners a modern, efficient, and
reliable solution where pet care, adoption, and automated feeding can be managed
through a single platform.

The website includes pet adoption listings, allowing users to browse available
pets and submit adoption requests directly. It also provides breed-based
information, health tips, vaccination reminders, and daily care guidelines. A
dedicated marketplace section lets users explore pet accessories such as collars,
feeders, toys, grooming tools, and other pet-care products. Secure user login,
appointment booking for veterinary or grooming services, and an intuitive
dashboard enhance the usability of the system.

A major innovation of this project is the IoT-based Smart Pet Feeder. Through the
website, users can set preferred feeding times and the exact quantity of food
required. The system automatically records feed logs and provides status updates
back to the platform, offering reliability, convenience, and consistent feeding
routines even when the owner is away.

By combining a well-structured web platform with IoT-controlled feeding, PetCloud
delivers a comprehensive solution for pet care, adoption support, accessory
browsing, and automated feeding.

  Backend   : PHP, MySQL Database

  Tools     : XAMPP, VS Code, Arduino IDE, Git

CHAPTER 1
INTRODUCTION


1.1 PROJECT OVERVIEW

PetCloud is a full-stack, multi-role web and mobile platform designed to serve as
a one-stop solution for modern pet owners in India. It was developed to address
the growing need for a centralised digital system that can handle the day-to-day
complexities of pet care, without forcing users to switch between multiple
disconnected apps or services.

The platform serves three distinct user roles:
  - Pet Owners (Clients): who manage their pets, book vet appointments, shop for
    pet products, report lost pets, and use the smart feeder control system.
  - Shop Owners: who register their pet stores on the platform and manage product
    listings, customer orders, and reviews through a dedicated shop portal.
  - Administrators: who oversee the entire platform, approve shop registrations,
    manage adoption listings, and monitor all transactions and user activity.

The key innovation of PetCloud is its Smart Feeder module, which enables pet
owners to schedule automated feeding times and remotely trigger their pet feeder
device through the web or mobile app. Feed logs are automatically recorded in the
database, and the last feed time and portion are displayed on the dashboard.

PetCloud also features a community-driven lost-and-found pet alert network. When
a pet is reported lost, the system notifies nearby users on their dashboards based
on city-level location matching. Community members can report sightings, and
owners can mark their pet as found once located.

The companion mobile application (PetCloud Mobile) is built using React Native
and Expo SDK 54, supporting both Android and iOS. It connects to the same backend
server through a RESTful mobile API layer housed at /mobile_api/, allowing all
core features of the web platform to be accessed from a smartphone.

The project was developed over approximately 16 weeks using XAMPP (Apache + PHP +
MySQL) for local hosting, with Git and GitHub for version control.

CHAPTER 2
SYSTEM STUDY


2.1 INTRODUCTION

The study of existing pet care systems reveals a significant gap between the
tools available to pet owners and what they actually need to manage their pets
effectively. Most current digital solutions are narrow in scope: a feeding app
that cannot book vet appointments, or a marketplace that does not track pet
health. This system study examines the natural and designed systems that
PetCloud was built to improve upon.

As per the study, the primary users of the system are:
  a) Pet owners who need a unified platform for all pet care tasks.
  b) Pet shop owners who need a digital storefront and order management system.
  c) Veterinary clinics that accept online appointment bookings.
  d) Community members who can help report lost or found pets.

The system study forms the basis for identifying what PetCloud must do, what
it must avoid repeating from older systems, and what new value it must provide.


2.2 EXISTING SYSTEM

2.2.1 NATURAL SYSTEM STUDIED

The natural (manual) pet care system observed involves:
  - Pet owners maintaining paper-based health records, vaccination cards, and
    handwritten feeding schedules for their pets.
  - Lost pet reports being shared via word-of-mouth, physical posters in the
    neighbourhood, and unstructured social media posts.
  - Pet adoption being arranged through personal contacts or informal listings
    at local veterinary clinics with no verification process.
  - Feeding done entirely manually with no automation or schedule tracking,
    leading to missed meals, overfeeding, or underfeeding.
  - Vet appointment booking done via phone calls, often resulting in
    double-bookings, long wait times, and missed reminders.

2.2.2 DESIGNED SYSTEM STUDIED

The following existing digital solutions were studied:
  a) PetMD (petmd.com): Provides basic pet health information and vet finder
     but lacks feeding management, marketplace, or adoption features.
  b) Petfinder (petfinder.com): Specialises in adoption listings but does not
     include health records, smart feeders, or shopping capabilities.
  c) Chewy (chewy.com): A large pet product e-commerce site with no community
     features, health records, or appointment booking integration.
  d) Generic IoT Feeder Apps (e.g., PetKit): Standalone smart feeder apps
     with no vet booking, no marketplace, and no adoption community.

None of the existing systems combine all the required features into one unified
platform with role-based access for pet owners, shop owners, and admins.


2.3 DRAWBACKS OF EXISTING SYSTEM

  * No single platform integrates pet health records, feeding management, vet
    appointment booking, marketplace shopping, and pet adoption together.
  * Lost pet reporting through social media has limited geographic targeting and
    no structured follow-up or sighting notification system.
  * Smart feeders in the market are standalone IoT devices without web
    dashboards, health tracking, or appointment integration.
  * Pet adoption processes on existing platforms lack an admin approval layer,
    making it easy for unreliable or unsafe adoptions to occur.
  * Shop owners who sell pet products have no dedicated, pet-focused digital
    storefront - they rely on general e-commerce platforms.
  * No existing system provides role-based portals for pet owners, shop owners,
    and administrators within the same pet-care ecosystem.
  * Health reminders and vaccination tracking are not integrated with appointment
    booking systems in any single existing platform.


2.4 PROPOSED SYSTEM

PetCloud is proposed as a unified, multi-role pet care platform that solves all
the drawbacks identified in the existing system study. The proposed system:

  a) Provides a single login portal for three distinct user roles: Pet Owner
     (client), Shop Owner, and Administrator, each with a dedicated dashboard.

  b) Implements a Smart Feeder module where pet owners can set scheduled feeding
     times and trigger instant manual feeds from the web or mobile app. Feed
     events are logged in the database (feed_logs table) with timestamps and
     portion data.

  c) Builds a community-driven lost-and-found pet alert network. When a pet is
     marked lost, other users in the same city see the alert on their dashboard.
     Community members can report sightings, and owners can mark pets as found.

  d) Provides a structured pet adoption system with admin approval workflows.
     Listings are verified before going public, ensuring safe and reliable
     pet placements.

  e) Integrates a marketplace for pet products where approved shop owners list
     items, and pet owners can add them to cart and check out using Razorpay
     payment gateway (supporting cards, net banking, wallets, and EMI).

  f) Offers a multi-step veterinary appointment booking system with 12 service
     categories, clinic selection, time-slot picking, and Razorpay payment.

  g) Provides a companion React Native mobile app (PetCloud Mobile) built on
     Expo SDK 54, connecting to the same PHP/MySQL backend through a
     dedicated /mobile_api/ REST layer.

  h) Uses PDO prepared statements for all database queries, bcrypt password
     hashing, and Google OAuth via Firebase for secure authentication.


2.5 ADVANTAGES OF PROPOSED SYSTEM

  * All pet care features are available on one platform - no need to juggle
    multiple apps for feeding, health, booking, shopping, and adoption.
  * The smart feeder module allows automated and remote feeding, ensuring
    pets are fed on time even when the owner is away.
  * Location-aware lost pet alerts reach nearby community members immediately,
    increasing the chance of finding a lost pet quickly.
  * Admin approval workflows for shops and adoption listings ensure platform
    quality and user safety.
  * Razorpay integration (cards, net banking, wallets, EMI) provides a smooth,
    trusted, and familiar payment experience for Indian users.
  * The React Native mobile app allows users to manage all features from their
    smartphone without needing a browser.
  * Role-based access control ensures each user type (client, shop, admin) only
    sees and can perform actions relevant to their role.
  * PDO prepared statements protect all database queries from SQL injection.
  * Google OAuth (Firebase) login reduces friction for new users signing up.
  * Feed logs give owners a historical record of when and how much their pet
    was fed, useful for monitoring pet health and nutrition.

The feasibility study evaluates whether the PetCloud project can be successfully
developed and deployed given the available resources, technology, budget, and
schedule. Four types of feasibility are examined below.

All core technologies used by PetCloud are free and open-source:
  - PHP, MySQL, Apache (XAMPP): Free
  - React Native and Expo SDK: Free
  - Firebase Authentication (Spark Plan): Free
  - PHPMailer: Free (open-source MIT license)

Infrastructure costs:

  - Production deployment (future): VPS hosting approx. Rs. 500-1500/month
  - Domain name (future): approx. Rs. 500-1000/year

VERDICT: The project is economically feasible as development incurs zero
licensing or infrastructure cost.


3.1.2 Technical Feasibility

The PetCloud system is built on well-established, widely-documented technologies:
  - PHP 8.x with PDO is a mature and stable backend technology with extensive
    community support, tutorials, and documentation.
  - MySQL 8.0 is an industry-standard relational database used in millions of
    production systems worldwide.
  - React Native (Expo SDK 54) allows a single JavaScript codebase to target
    both Android and iOS, reducing mobile development effort significantly.
  - XAMPP provides a one-click local development environment for Windows.
  - Razorpay and Firebase have official, well-maintained SDKs for PHP and JS.

available within the development team.

VERDICT: The project is technically feasible.

PetCloud's user interface follows modern web and mobile design conventions that
smartphone and computer users are already familiar with:

    Practo, Swiggy, and other popular Indian apps.
  - Role-based portals give each user type a clean, focused experience without
    overwhelming them with irrelevant features.
  - No specialised technical knowledge is required to use the platform.

VERDICT: The project is behaviorally feasible.

3.3 SOFTWARE DESCRIPTION

3.3.1 PHP (Hypertext Preprocessor)

PHP is a widely-used, open-source, server-side scripting language designed
primarily for web development. In PetCloud, PHP 8.x is used to:
  - Handle all HTTP requests from the browser and mobile app
  - Connect to the MySQL database using PDO (PHP Data Objects)
  - Manage user sessions for authentication and role-based redirection
  - Process form submissions, file uploads (pet photos, product images)
  - Serve as the REST API backend for the React Native mobile app
    (all endpoints in /mobile_api/ folder return JSON responses)
  - Send transactional emails using PHPMailer (password reset links)
  - Integrate with Razorpay payment gateway for appointment and order payments

Key PHP security practices used in PetCloud:
  - All database queries use PDO prepared statements (prevents SQL injection)
  - Passwords hashed with password_hash() using bcrypt algorithm
  - Output sanitised with htmlspecialchars() (prevents XSS attacks)
  - Sessions managed with session_start() and session_regenerate_id()


3.3.2 MySQL

MySQL is a popular open-source relational database management system (RDBMS).
In PetCloud, MySQL 8.0 is used via the PDO extension in PHP.

The database (petcloud_db) contains 25+ tables:
  users, user_pets, feeding_schedules, feed_logs, smart_feeder_schedules,
  appointments, hospitals, health_reminders, adoption_listings,
  adoption_applications, lost_pet_alerts, found_pet_reports,
  general_found_pets, pet_rehoming_listings, orders, products, cart,
  shops, shop_applications, services, service_categories, daily_tasks,
  admin_activity_logs, password_resets, product_reviews

Key MySQL features used:
  - Foreign key relationships between tables (e.g., user_pets.user_id
    references users.id)
  - Indexed columns for faster lookups (user_id, pet_id, status)
  - DATETIME columns with IST timezone (SET time_zone = '+5:30')
  - Transactions for multi-step operations (appointment + payment)

4.3 DATABASE DESIGN

4.3.1 Relational Database Management System (RDBMS)

What is an RDBMS?
  A Relational Database Management System (RDBMS) is software that stores and
  manages data in a structured, tabular format using rows and columns. Data is
  organised into tables (also called relations), and relationships between tables
  are established using primary keys and foreign keys. RDBMS systems support the
  Structured Query Language (SQL) for defining, querying, and manipulating data.

  Core principles of an RDBMS:
    - Data is stored in tables (relations) with rows (records) and columns (fields)
    - Each table has a PRIMARY KEY that uniquely identifies every row
    - Tables are linked via FOREIGN KEYS that reference primary keys in other tables
    - The ACID properties (Atomicity, Consistency, Isolation, Durability) ensure
      data integrity and reliability even during failures or concurrent access
    - Referential integrity enforces that a foreign key value must match an
      existing primary key in the referenced table

Why MySQL for PetCloud?
  MySQL 8.0 was selected as the RDBMS for PetCloud for the following reasons:
    * Open-source and free to use (GNU General Public License)
    * Industry-standard RDBMS used in millions of web applications worldwide
    * Fully integrated with XAMPP (Apache + PHP + MySQL bundle for Windows)
    * Excellent performance for read-heavy web applications with proper indexing
    * Native support for foreign keys, transactions, and cascading operations
    * Managed easily through phpMyAdmin (a GUI tool included in XAMPP)
    * Widely supported by PHP through the PDO (PHP Data Objects) extension
    * Strong community support, documentation, and tutorials

How PetCloud Uses MySQL:
  Database Name   : petcloud_db
  Total Tables    : 25+
  Connection      : PHP PDO (db_connect.php)
  Access Method   : Prepared Statements (100% SQL-injection safe)
  Timezone        : Asia/Kolkata (IST, UTC+5:30) — set at connection time
  Character Set   : utf8mb4 (supports all Unicode characters including emojis)

  PetCloud's MySQL database is structured around 6 core domains:

    1. Users & Authentication
       Tables: users, password_resets, admin_activity_logs
       Purpose: Store user accounts, reset tokens, and admin audit trails

    2. Pet Management
       Tables: user_pets, feeding_schedules, feed_logs, smart_feeder_schedules,
               health_reminders, daily_tasks
       Purpose: Pet profiles, feeding automation, health tracking

    3. Veterinary & Appointments
       Tables: appointments, hospitals, services, service_categories
       Purpose: Multi-step booking system, clinic registry, service catalogue

    4. Community Features
       Tables: lost_pet_alerts, found_pet_reports, general_found_pets,
               adoption_listings, adoption_applications, pet_rehoming_listings
       Purpose: Lost pet network, peer-to-peer adoption and rehoming

    5. Marketplace & Commerce
       Tables: products, cart, orders, product_reviews
       Purpose: Pet product shopping, cart management, order tracking

    6. Shops & Platform Management
       Tables: shops, shop_applications
       Purpose: Shop owner registration, approval, and management

  Key MySQL Features Used in PetCloud:
    - AUTO_INCREMENT primary keys on all tables for unique ID generation
    - FOREIGN KEY constraints with ON DELETE CASCADE (e.g., deleting a user
      removes all their pets, schedules, and alerts automatically)
    - ENUM data types for status fields (e.g., 'Active'/'Lost', 'Pending'/
      'Confirmed'/'Cancelled', 'client'/'shop_owner'/'admin')
    - DATETIME columns for all timestamps, stored in IST timezone
    - DECIMAL(10,2) for all monetary values (prices, order totals, costs)
    - TEXT columns for long descriptions (product descriptions, reasons)
    - VARCHAR with appropriate lengths for name, email, address fields

      the payment and booking are saved together or not at all

4.3.2 Normalization

Normalization is the process of organizing a relational database to reduce
data redundancy and improve data integrity by dividing large tables into
smaller, well-structured tables connected through relationships.

PetCloud's database is normalized to Third Normal Form (3NF).

First Normal Form (1NF):
  Rules:
    - Every column must contain atomic (indivisible) values
    - Each column must contain values of a single type
    - Each row must be uniquely identifiable (primary key required)
    - No repeating groups or arrays in a single column

  PetCloud Example - BEFORE 1NF (violation):
    users table: id | name      | pets
                 1  | Joshua    | Bruno, Max, Luna    ← multiple values in one column

  PetCloud Example - AFTER 1NF (correct):
    users table  : id | name
                   1  | Joshua

    user_pets table: id | user_id | pet_name
                     1  | 1       | Bruno
                     2  | 1       | Max
                     3  | 1       | Luna

  All PetCloud tables satisfy 1NF: each column stores one atomic value,
  and every table has an AUTO_INCREMENT primary key (id).

Second Normal Form (2NF):
  Rules:
    - Must already be in 1NF
    - No partial dependency: every non-key attribute must depend on the
      entire primary key (applies to composite primary keys)

  PetCloud Example - BEFORE 2NF (violation):
    Suppose a composite key (user_id, pet_id) in a feeding table:
    user_id | pet_id | user_email        | feeding_time
    1       | 5      | josh@example.com  | 08:00
    Here, user_email depends only on user_id (partial dependency) ← violation

  PetCloud Example - AFTER 2NF (correct):
    feeding_schedules: id | user_id | pet_id | feeding_time
    users            : id | email   | ...
    user_email is stored only in users, referenced via user_id (FK).

  All PetCloud tables use a single-column surrogate primary key (id),
  so 2NF violations are structurally impossible.

Third Normal Form (3NF):
  Rules:
    - Must already be in 2NF
    - No transitive dependency: non-key attributes must not depend on
      other non-key attributes

  PetCloud Example - BEFORE 3NF (violation):
    appointments: id | hospital_id | hospital_name | hospital_city
    Here, hospital_name and hospital_city depend on hospital_id (a non-key
    attribute), not on the primary key id. ← transitive dependency

  PetCloud Example - AFTER 3NF (correct):
    appointments: id | hospital_id (FK)
    hospitals   : id | name | city | address | phone
    hospital_name and hospital_city are stored only in hospitals table
    and accessed via the hospital_id foreign key.

  All PetCloud related data (hospital info, shop info, user info) is
  stored in its own table and linked via foreign keys, achieving 3NF.

Summary of Normalization in PetCloud:
  +-------+--------------+-----------------------------------------------+
  | Form  | Rule         | How PetCloud Satisfies It                    |
  +-------+--------------+-----------------------------------------------+
  | 1NF   | Atomic values| Each column stores one value; PKs on all tbls|
  | 2NF   | No partial   | Single-column surrogate PKs on all tables    |
  |       | dependency   | prevent partial dependencies                 |
  | 3NF   | No transitive| Hospital, shop, user data in separate tables |
  |       | dependency   | linked via foreign keys (not repeated cols)  |
  +-------+--------------+-----------------------------------------------+


4.3.3 Sanitization

Sanitization is the process of cleaning and validating user input to prevent
security vulnerabilities such as SQL Injection, Cross-Site Scripting (XSS),
and data corruption before it is processed or stored in the database.

PetCloud applies sanitization at three layers:

Layer 1: JavaScript Client-Side Validation (js/form-validation.js)
  - Validates required fields before form submission (prevents empty data)
  - Checks email format using regex: /^[^\s@]+@[^\s@]+\.[^\s@]+$/
  - Enforces minimum password length (8 characters)
  - Provides instant user feedback without a server round-trip

  Example (signup form validation):
    if (emailField.value.trim() === '') {
        showError(emailField, 'Email is required');
        return false;
    }

Layer 2: PHP Server-Side Sanitization
  All data received from POST/GET is sanitized before use:

  a) trim() — removes accidental leading/trailing whitespace
       $email = trim($_POST['email'] ?? '');
       $name  = trim($_POST['full_name'] ?? '');

  b) htmlspecialchars() — converts HTML special characters to safe entities,
     preventing XSS attacks when data is output back to the browser
       echo htmlspecialchars($user['full_name'], ENT_QUOTES, 'UTF-8');
       // Converts: <script>alert('xss')</script>
       // To safe:  &lt;script&gt;alert(&#039;xss&#039;)&lt;/script&gt;

  c) intval() / (int) casting — ensures numeric fields contain only integers
       $user_id = (int)($_SESSION['user_id']);
       $pet_id  = intval($_POST['pet_id'] ?? 0);

  d) Null coalescing (??) — safely handles missing POST parameters
       $quantity = $_POST['quantity'] ?? 'Medium (60g)';

Layer 3: PDO Prepared Statements (SQL Injection Prevention)
  All database queries use PDO prepared statements with parameter binding,
  which completely prevents SQL injection attacks by separating SQL code
  from user-supplied data.

  UNSAFE (raw SQL — NOT used in PetCloud):
    $result = $pdo->query("SELECT * FROM users WHERE email = '$email'");
    // If $email = "' OR 1=1 --", this dumps the entire users table!

  SAFE (prepared statement — used in all PetCloud queries):
    $stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?");
    $stmt->execute([$email]);
    // User input is bound as a data value, never executed as SQL code

  All 25+ tables in PetCloud are accessed exclusively through PDO
  prepared statements — no raw string concatenation in SQL queries.

Additional Security Measures:
  - Password hashing: password_hash($password, PASSWORD_BCRYPT) before storage
  - Password verification: password_verify($input, $stored_hash) on login
  - Session security: session_regenerate_id(true) after successful login
    to prevent session fixation attacks
  - File uploads: MIME type validation + extension whitelist for pet/product
    photos (only jpg, jpeg, png, gif allowed)


4.3.4 Indexing

Indexing is a database optimization technique that creates a special data
structure (a B-Tree index) on one or more columns to allow MySQL to locate
rows much faster without scanning the entire table.

How Indexing Works:
  Without an index, MySQL performs a Full Table Scan — it reads every row
  in the table until matching rows are found. This is slow for large tables.
  With an index, MySQL uses the B-Tree structure to jump directly to the
  matching rows in O(log n) time, making queries significantly faster.

  Example: Without index on users.email (10,000 rows):
    SELECT * FROM users WHERE email = 'test@petcloud.com';
    → MySQL scans all 10,000 rows                    O(n) — SLOW

  With index on users.email:
    → MySQL jumps directly to the match via B-Tree    O(log n) — FAST

Types of Indexes Used in PetCloud:

  1. PRIMARY KEY Index (automatic on all tables):
     MySQL automatically creates a clustered B-Tree index on the primary key
     column (id) of every table. This is the fastest lookup in PetCloud.

     Affected tables: ALL 25+ tables (id column)
     Query example:
       SELECT * FROM appointments WHERE id = 12;  ← instant lookup via PK index

  2. UNIQUE Index:
     Ensures no two rows have the same value, and provides fast lookup.

     users.email — UNIQUE index ensures no duplicate accounts:
       CREATE UNIQUE INDEX idx_users_email ON users(email);

  3. FOREIGN KEY Index (created by MySQL for FK columns):
     MySQL automatically indexes foreign key columns to speed up JOIN
     operations and referential integrity checks.

     Key foreign key indexes in PetCloud:
     +---------------------------+------------------+----------------------------+
     | Table                     | FK Column        | References                 |
     +---------------------------+------------------+----------------------------+
     | user_pets                 | user_id          | users(id)                  |
     | feeding_schedules         | user_id, pet_id  | users(id), user_pets(id)   |
     | feed_logs                 | user_id, pet_id  | users(id), user_pets(id)   |
     | appointments              | user_id          | users(id)                  |
     | appointments              | hospital_id      | hospitals(id)              |
     | orders                    | user_id          | users(id)                  |
     | products                  | shop_id          | shops(id)                  |
     | lost_pet_alerts           | user_id, pet_id  | users(id), user_pets(id)   |
     | adoption_listings         | user_id          | users(id)                  |
     | adoption_applications     | listing_id       | adoption_listings(id)      |
     | found_pet_reports         | alert_id         | lost_pet_alerts(id)        |
     | health_reminders          | user_id, pet_id  | users(id), user_pets(id)   |
     | cart                      | user_id          | users(id)                  |
     | shops                     | user_id          | users(id)                  |
     +---------------------------+------------------+----------------------------+

  4. Status Column Index (for filtered dashboard queries):
     The dashboard frequently filters by status (Active/Lost, Pending/Confirmed).
     Indexing status columns improves these queries:

     Index on lost_pet_alerts(status):
       SELECT * FROM lost_pet_alerts WHERE status = 'Active' AND city = ?
       → Without index: full table scan of all alerts
       → With index: only 'Active' rows scanned (much smaller set)

     Index on appointments(status):
       SELECT * FROM appointments WHERE user_id = ? AND status = 'Confirmed'

Impact of Indexing on PetCloud Performance:
  +----------------------------------+------------------+------------------+
  | Query                            | Without Index    | With Index       |
  +----------------------------------+------------------+------------------+
  | Login by email lookup            | Full table scan  | O(log n) — fast  |
  | Dashboard: pets by user_id       | Full table scan  | FK index used    |
  | Apt list: appointments by user   | Full table scan  | FK index used    |
  | Lost alerts: filter by status    | Full table scan  | Status index used|
  | Products: filter by shop_id      | Full table scan  | FK index used    |
  +----------------------------------+------------------+------------------+

System testing is the process of verifying that the developed software behaves
as expected and meets all specified requirements. Testing was performed at
multiple levels for PetCloud: unit testing of individual PHP API endpoints,

5.2.5 Automation Testing (Selenium WebDriver)

What is Automation Testing?
  Automation Testing is the process of using software tools to automatically
  execute predefined test cases, compare actual results with expected outcomes,
  and generate test reports — without manual intervention. It is used to:
    - Speed up regression testing by running the same tests repeatedly
    - Ensure consistent and unbiased test execution
    - Free testers from repetitive manual checks
    - Run tests on every code change to catch bugs early

  For PetCloud, Selenium WebDriver with Python was used to automate the UI
  testing of critical user-facing forms and flows on the web application.

Tool Used: Selenium WebDriver
  Selenium WebDriver is an open-source browser automation framework. It allows
  test scripts to interact with web pages exactly as a real user would — clicking
  buttons, filling forms, navigating pages, and validating results.

5.2.6 Selenium Testing

What is Selenium?
  Selenium is a free, open-source suite of tools designed specifically for
  automating web browser interactions. It is the most widely used browser
  automation framework in the world, supported by all major browsers
  (Chrome, Firefox, Edge, Safari) and all major programming languages
  (Python, Java, C#, JavaScript).

  Selenium was originally created by Jason Huggins in 2004 at ThoughtWorks
  and is now maintained by the Selenium community under an open-source license.

Selenium Suite Components:

  1. Selenium WebDriver
     The core component used in PetCloud testing. WebDriver provides a
     programming interface to interact with a real browser. It communicates
     directly with the browser using the browser's native automation protocol
     (W3C WebDriver standard), making it faster and more reliable than
     older Selenium RC.

  2. Selenium Grid
     Allows tests to run in parallel across multiple machines and browsers
     simultaneously. Not used in PetCloud (single-machine development), but
     can be added for future CI/CD pipeline testing.

  3. Selenium IDE
     A browser extension (Chrome/Firefox) that records and replays browser
     interactions without writing code. Used for quick exploratory testing
     during early PetCloud development phases.

How Selenium WebDriver Works:

  Test Script (Python)
        |
        | Selenium WebDriver API calls
        v
  WebDriver (ChromeDriver)
        |
        | W3C WebDriver Protocol (HTTP/JSON)
        v
  Google Chrome Browser
        |
        | Renders Pages, Executes JS
        v
  PetCloud Web App (localhost/PetCloud)
        |
        | HTTP Requests
        v
  Apache + PHP + MySQL (XAMPP)

  Step-by-step execution:
    1. Test script calls driver.get(URL) → ChromeDriver opens Chrome
    2. Chrome navigates to the PetCloud URL
    3. Script calls find_element() → ChromeDriver locates DOM element
    4. Script calls send_keys() or click() → action performed in browser
    5. WebDriverWait polls the DOM until a condition is met (e.g. URL changes)
    6. assert statement verifies the outcome
    7. driver.quit() closes the browser window

Advantages of Selenium for PetCloud:
  * Free and open-source — no licensing cost
  * Tests run against a real Chrome browser — no simulation
  * Supports dynamic web elements (modals, AJAX, dropdowns)
  * execute_script() allows JS injection to handle stubborn modal forms
  * WebDriverWait with Expected Conditions prevents flaky timing issues
  * webdriver-manager auto-keeps ChromeDriver in sync with Chrome version
  * Test results are immediately visible in the terminal

Limitations of Selenium in PetCloud Context:
  * Requires XAMPP to be running — tests fail if server is offline
  * Tests run sequentially (not parallel) without Selenium Grid
  * Browser UI opens for each test — no headless mode currently configured
    (--headless flag can be added to ChromeOptions to run invisibly)
  * Mobile app (React Native / Expo) cannot be tested with Selenium
    — Appium would be needed for mobile automation
  * Google OAuth login cannot be automated via Selenium easily due to
    Google's bot-detection on login pages

  - Monitor feed_logs and admin_activity_logs tables for unusual activity

const SmartFeederScreen = () => {
  const [selectedPet, setSelectedPet] = useState(null);
  const [portion, setPortion] = useState('Medium (60g)');
  const [lastFeed, setLastFeed] = useState(null);

  const triggerFeed = async () => {
    const res = await fetch(${API_URL}/trigger_manual_feed.php, {
      method: 'POST',
      headers: {'Content-Type': 'application/json'},
      body: JSON.stringify({
        user_id: userId,
        pet_id: selectedPet.id,
        quantity: portion
      })
    });
    const json = await res.json();
    if (json.ok) {
      setLastFeed(json.fed_at);
      Alert.alert('Success', 'Your pet has been fed!');
    }
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>Smart Feeder</Text>
      <TouchableOpacity onPress={triggerFeed} style={styles.feedBtn}>
        <Text style={styles.btnText}>Feed Now</Text>
      </TouchableOpacity>
      {lastFeed && <Text>Last fed: {lastFeed}</Text>}
    </View>
  );
};
